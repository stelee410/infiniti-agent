# Infiniti Agent Code Review Guidelines

本规范用于 `infiniti-agent` 项目的专项 code review，重点识别反复修改仍未收敛的问题、重复代码、冗余实现、模块边界混乱和缺少测试证据的风险。

## 1. Review 目标

本轮 review 不追求一次性重构所有代码，而是优先找出会持续拖慢开发、反复引发缺陷或增加维护成本的问题。

核心目标：

- 识别重复出现的实现模式、临时补丁和相互覆盖的逻辑。
- 找出可合并、可抽象或应删除的冗余代码。
- 检查关键模块的边界是否清晰，避免 UI、LLM、工具执行、持久化、配置、音视频能力互相泄漏。
- 确认重要行为有测试覆盖，尤其是此前反复修改过的逻辑。
- 输出可执行的问题清单，而不是泛泛的风格意见。

## 2. Review 范围

优先范围：

- `src/llm/`：消息构造、压缩、持久化、run loop、工具调用门控。
- `src/tools/`：工具定义、执行器、repo/workspace 访问、文本 diff。
- `src/memory/` 和 `src/subconscious/`：长期记忆、结构化记忆、consolidation 流程。
- `src/liveui/` 和 `liveui/`：实时 UI 协议、渲染进程、语音/表情/模型路径处理。
- `src/tts/`、`src/asr/`、`src/avatar/`、`src/video/`：外部模型、文件生成、异步任务和错误处理。
- `src/config/`、`src/paths.ts`、`src/packageRoot.ts`：配置读取、路径解析、环境差异。
- `src/session/`、`src/inbox/`、`src/schedule/`：本地状态、归档、调度和数据一致性。

暂不作为主线的问题：

- 单纯格式化偏好。
- 尚未被调用的实验目录，除非它污染构建、发布或运行路径。
- 大型架构重写建议，除非当前缺陷无法用小步修复。

## 3. 问题分级

使用 P0 到 P3 标记问题严重度。

- **P0 阻断级**：会导致数据丢失、命令误执行、凭据泄漏、构建完全失败或核心功能不可用。
- **P1 高风险**：常见路径会崩溃、状态不一致、重复执行外部副作用、测试明显覆盖不到关键分支。
- **P2 中风险**：重复代码、冗余逻辑、模块职责混乱、错误处理不完整，短期可运行但会持续放大维护成本。
- **P3 改进项**：命名、局部可读性、轻微抽象机会、非关键测试补充。

每个问题必须说明：

- 具体文件和行号。
- 可复现或可推导的风险。
- 为什么当前实现会出问题。
- 建议修复方向。
- 是否需要测试补充。

## 4. 重复修改问题的专项标准

对近期反复改动或明显被补丁覆盖的逻辑，按下面标准审查：

- 是否存在多个入口在处理同一业务规则。
- 是否有相似的 fallback、retry、path resolve、env parse、JSON parse、stream parse、message transform 逻辑散落在多个文件。
- 是否存在“先特殊判断，再由下游重新特殊判断”的双重逻辑。
- 是否有测试只覆盖最终结果，没有覆盖导致反复修改的边界条件。
- 是否有旧实现保留但不再被调用，或新旧实现同时可达。
- 是否用注释、配置或魔法常量绕过了真正的边界问题。

反复修改问题的结论优先落在以下三类：

- **Consolidate**：合并重复入口或共享 helper。
- **Delete**：删除旧路径、死代码或不再需要的兼容分支。
- **Guard**：保留多路径，但用类型、测试或明确的 runtime guard 约束行为。

## 5. 重复代码与冗余代码标准

重复代码不只看文本相似，还看语义相似。

需要标记的问题：

- 相同输入清洗、路径展开、错误格式化、日志拼接、消息裁剪逻辑重复出现。
- 多个模块各自维护等价类型、协议字段或默认配置。
- 异步任务、文件写入、外部 API 调用缺少统一取消、重试、超时或错误包装策略。
- 测试里复制大量生产逻辑，导致测试与实现一起错。
- 分支逻辑永远不可达，或仅服务于已删除的旧配置。

不应机械抽象的问题：

- 两处代码只是表面相似，但属于不同领域规则。
- 抽象后会把 UI、CLI、LLM、工具执行或持久化边界混在一起。
- 抽象需要引入全局状态或让依赖方向倒置。

## 6. 模块边界标准

审查时按依赖方向判断代码是否健康。

- CLI/UI 负责交互，不应直接承担核心业务规则。
- `src/llm/` 负责模型消息、工具调用编排和上下文管理，不应知道 live UI 或具体音视频实现细节。
- `src/tools/` 负责工具声明和执行，不应直接修改 UI 状态。
- `src/memory/` 和 `src/subconscious/` 负责记忆模型与生命周期，不应依赖终端渲染或临时命令输出格式。
- `src/config/` 和路径模块应成为配置、环境变量、工作区路径的单一入口。
- live UI 进程通信必须以协议类型为准，避免字符串事件名和 payload 结构散落。

发现跨边界依赖时，优先建议：

- 将纯逻辑下沉到可测试 helper。
- 用协议类型、接口或 options 参数隔离调用方。
- 将副作用集中到少数边界模块。

## 7. TypeScript 与错误处理标准

重点检查：

- 是否使用 `any`、宽泛 object 或字符串枚举绕开类型约束。
- 外部输入是否经过 schema、类型守卫或显式验证。
- async 函数是否吞掉错误，或只 `console.error` 不向上报告。
- 文件、网络、模型调用是否有超时、取消、重试边界。
- 错误信息是否包含排查所需上下文，同时避免泄漏 token、路径中敏感信息或用户内容。
- 默认值是否集中定义，避免在多个文件用不同默认值。

## 8. 测试标准

每个非 P3 问题都要判断是否需要测试。

优先补充测试的场景：

- 反复修改过的逻辑。
- 消息压缩、工具调用、路径解析、归档、记忆合并、调度解析。
- 文件系统写入、外部命令、模型调用的边界条件。
- 兼容旧数据结构或配置迁移。

测试要求：

- 优先使用现有 Vitest 风格。
- 断言行为，不复制实现。
- 对 bugfix 增加失败用例，再验证修复。
- 对外部 API、模型和系统命令使用 mock 或薄封装，不依赖真实服务。

常用验证命令：

```bash
npm test
npm run build
npm test -- src/path/to/file.test.ts
```

## 9. Review 输出格式

每个发现按以下格式输出：

```md
### [P1] 问题标题

- 位置：`src/example.ts:42`
- 类型：重复代码 / 冗余代码 / 边界混乱 / 错误处理 / 测试缺口 / 其他
- 现象：当前代码如何表现。
- 风险：为什么会造成 bug、重复修改或维护负担。
- 建议：推荐修复方向。
- 测试：需要补充或运行的测试。
```

最终汇总必须包含：

- P0/P1/P2/P3 数量。
- 重复代码热点列表。
- 建议优先修复顺序。
- 可直接删除的冗余代码清单。
- 需要补测试的模块清单。
- 不建议立即重构但需要观察的区域。

## 10. Review 原则

- 先找风险，再谈风格。
- 先证明问题，再建议改法。
- 小步可落地优先于大而全重构。
- 不因为代码“看起来重复”就抽象，要确认重复的是业务语义。
- 不把实验性能力和核心路径混为一谈。
- 对反复修改的问题，优先追问“为什么之前没有收敛”。
- 任何建议都应能转化为 issue、PR 或测试用例。

## 11. 代码规范与模块化专项 Review

本专项用于第二轮 code review，重点不再是安全漏洞，而是代码是否清晰、职责是否单一、重复逻辑是否收敛、文件和函数是否过大、模块是否容易继续演进。

### 11.1 Review 目标

本轮 review 的核心目标：

- 找出重复的业务逻辑、重复的工具函数、重复的状态流转。
- 找出过大的文件、过长函数、深层嵌套和难以测试的代码块。
- 判断功能是否拆在正确模块中，避免“哪里方便就写哪里”。
- 找出命名、类型、参数形态不清晰导致的维护成本。
- 给出可分阶段落地的重构建议，避免一次性大改导致回归。

### 11.2 重复逻辑判断标准

重复逻辑按“语义重复”判断，不只看文本是否相似。

需要标记的问题：

- 多个文件各自实现相同的 path resolve、env parse、config fallback、JSON validate、log append、session append、job enqueue、HTTP timeout、retry、stream parse。
- 多个函数接受不同参数名，但实际做同一件事。
- 同一功能存在 CLI 入口、UI 入口、工具入口三套实现，且没有共享核心函数。
- 同一状态字段在多个模块里各自推导，例如 renderer、voice、inbox、memory、schedule 的状态文本或默认值。
- 同一协议 payload 在前后端各写一份字符串判断，没有共享类型或校验函数。
- 测试中复制生产逻辑，导致测试只能证明“复制粘贴一致”，不能证明行为正确。

重复逻辑的处理优先级：

- **必须合并**：涉及副作用、权限、安全、文件写入、网络请求、会话状态、异步任务生命周期。
- **建议合并**：格式化、日志、错误消息、轻量转换、默认值。
- **可以保留**：两处代码只是形状相似，但业务含义和变化原因不同。

### 11.3 文件大小标准

文件大小不是机械限制，但过大文件必须解释“为什么不能拆”。

建议标准：

- **0-250 行**：健康范围。
- **250-500 行**：需要确认职责是否单一。
- **500-800 行**：应作为 review 关注点，优先找可拆分区域。
- **800 行以上**：默认标记为结构风险，除非是声明表、生成文件或非常稳定的纯数据文件。
- **1000 行以上**：必须提出拆分方案，即使暂不立即执行。

当前项目优先关注的大文件类型：

- UI 容器文件，例如 `ChatApp.tsx`、`liveui/src/main.ts`。
- 编排型文件，例如 `src/llm/runLoop.ts`、`src/tools/runner.ts`。
- 配置解析文件，例如 `src/config/io.ts`、`src/config/types.ts`。
- 协议与适配层，例如 `src/liveui/wsSession.ts`、`src/liveui/protocol.ts`。

拆分文件时优先按职责拆：

- 纯类型：`types.ts`、`protocol.ts`。
- 纯解析：`parse*.ts`、`validate*.ts`。
- 纯格式化：`format*.ts`。
- 副作用边界：`store.ts`、`client.ts`、`runner.ts`、`transport.ts`。
- UI 状态与 UI 渲染分离：hooks/state 与 components 分开。

### 11.4 函数大小标准

函数 review 不只看行数，还看认知负担。

建议标准：

- **0-40 行**：通常健康。
- **40-80 行**：需要确认是否只有一个职责。
- **80-150 行**：应检查是否能拆出 parse、validate、format、side-effect helper。
- **150 行以上**：默认标记为结构风险。

需要拆函数的信号：

- 一个函数里同时做输入解析、权限判断、业务执行、持久化、UI 通知、日志。
- `if/else` 或 `switch` 分支超过 5 个，且每个分支都有副作用。
- `try/catch/finally` 嵌套多层，错误语义难以判断。
- 函数内部出现多个“阶段性注释”，说明它已经承担多个步骤。
- 局部变量需要跨很长距离被读取或修改。
- 单元测试只能测端到端，无法单独测核心分支。

推荐拆法：

- `parseInput`：只处理输入转换，不做副作用。
- `validateInput`：只返回错误或 typed value。
- `buildRequest` / `buildMessage`：只构造请求对象。
- `execute*`：只执行核心业务副作用。
- `formatResult`：只格式化展示或返回值。
- `persist*`：只负责写入存储。

### 11.5 模块边界标准

模块要有清晰所有权。review 时先问：这段逻辑“属于谁”？

边界规则：

- `src/cli*` 只负责命令解析、参数适配、进程退出码，不承载核心业务。
- `src/ui/` 只负责交互状态和展示，不直接实现工具、memory、schedule、LLM 核心逻辑。
- `src/llm/` 只负责模型消息、provider 适配、工具调用编排，不直接理解具体业务工具的内部实现。
- `src/tools/` 负责工具声明、参数转换、执行分发；具体领域逻辑应下沉到对应模块。
- `src/session/` 负责会话读写、归档、回滚，不应散落到异步 worker 或 UI 里。
- `src/jobs/` 或类似模块负责异步任务通用生命周期，不应由 snap/video/avatar 各自复制。
- `src/config/` 负责配置 schema、默认值、迁移和读写，不应由 UI 面板自由拼装未校验对象。
- `src/liveui/` 负责 Node 与渲染进程协议边界；渲染端具体布局逻辑应留在 `liveui/src/`。

违反边界的常见表现：

- UI 文件直接读写底层 JSON store。
- provider 分支里夹带具体工具业务逻辑。
- CLI 命令复制业务模块已有实现。
- 配置默认值在 UI、CLI、config parser 三处同时出现。
- 同一错误消息由多个模块各自拼接。

### 11.6 抽象标准

抽象要服务于收敛变化点，而不是追求“看起来更高级”。

应该抽象：

- 同类副作用生命周期重复三次以上。
- 修 bug 时必须改多个文件才能保持一致。
- 测试需要在多个模块重复 mock 同一套基础设施。
- 参数和返回值已经自然形成稳定协议。
- 重复逻辑包含安全、权限、持久化、并发、网络、文件系统等高风险行为。

不应该抽象：

- 只有两处短代码相似，但业务变化方向不同。
- 抽象后出现 `mode`、`kind`、`type` 分支爆炸。
- 抽象要求上层传入大量回调，导致调用方比原来更难读。
- 抽象把 UI、LLM、工具、存储的依赖方向混在一起。

抽象结果必须满足：

- 名字能准确表达职责。
- 参数数量少且类型明确。
- 单元测试能直接覆盖抽象出来的核心逻辑。
- 调用方代码比原来更短、更清楚。

### 11.7 命名与 API 标准

命名必须让读者不跳转也能猜到行为。

需要标记的问题：

- 函数名只描述技术动作，不描述业务意图，例如 `handleThing`、`processData`、`runStuff`。
- 布尔参数含义不清，例如 `run(x, true, false)`。
- 函数返回 JSON 字符串但名字看起来像返回对象。
- `load`、`read`、`get` 混用，无法判断是否有副作用。
- `save`、`write`、`persist` 混用，无法判断是否覆盖、追加、合并。
- 类型名和运行时协议名不一致。

推荐约定：

- `parse*`：字符串或 unknown 到结构化对象，失败可返回 null 或抛明确错误。
- `validate*`：只验证并收窄类型，不做业务执行。
- `resolve*`：根据配置、路径、环境推导最终值。
- `build*`：构造请求、消息、payload，不做 IO。
- `load*` / `read*`：读取，不修改。
- `save*` / `write*`：覆盖写入。
- `append*`：追加写入。
- `execute*` / `run*`：有副作用。
- `format*`：展示文本或结构化输出。

### 11.8 测试与重构标准

结构性重构必须先有测试保护。

重构前测试要求：

- 对要抽取的纯函数补单元测试。
- 对已有 bug 或重复修改点补回归测试。
- 对文件写入、异步任务、协议解析使用小范围 mock 或临时目录。
- 对 UI 大文件拆分，至少保留关键状态转换或协议事件测试。

重构后验证：

```bash
npm test
npm run build
npm test -- src/path/to/changed.test.ts
```

如果只做结构拆分，测试断言应保持行为不变。

### 11.9 输出格式

代码规范专项 review 的每个发现按以下格式输出：

```md
### [P2] 问题标题

- 位置：`src/example.ts:120`
- 类型：重复逻辑 / 过大文件 / 过长函数 / 模块边界 / 命名不清 / 抽象缺失 / 过度抽象
- 现象：当前代码如何组织。
- 影响：为什么会让后续修改变慢、变危险或难测试。
- 建议：如何拆分、合并、改名或移动。
- 落地顺序：先补测试 / 先抽 helper / 再迁移调用方 / 最后删除旧路径。
```

最终报告必须包含：

- 过大文件排行榜。
- 过长函数/代码块清单。
- 重复逻辑热点。
- 建议抽象出的模块或 helper。
- 不建议立即抽象的区域。
- 推荐的分阶段重构计划。
